What is an FLA?

An FLA file (Adobe Animate / Flash CS5+) is just a ZIP file with a different extension. Rename foo.fla to foo.zip and open it — you'll see XML everywhere.

Old vs new: Pre-CS5 FLAs use a proprietary binary format (magic bytes D0 CF 11 E0 — that's a compound document / OLE2 container). Only CS5+ FLAs are ZIP-based and parseable as XML. The inspector checks magic[:2] == b'PK' to detect ZIP.

Inside the ZIP you'll find two kinds of files:

  • DOMDocument.xml — one file, the stage settings + master symbol list
  • LIBRARY/SymbolName.xml — one file per library symbol, containing all its art

Everything is in the XFL namespace: http://ns.adobe.com/xfl/2008/. Every tag you'll encounter is prefixed with that URI in the parsed tree, which is why the inspector wraps lookups with t('TagName') → {http://ns.adobe.com/xfl/2008/}TagName.

Other files in the ZIP (skip these)
FilePurpose
mimetypeLiteral string "application/vnd.adobe.fla" — not XML
META-INF/metadata.xmlUsually empty in our FLAs
PublishSettings.xmlExport/publish configuration (target SWF version etc.)
MobileSettings.xmlMobile publish settings
bin/*.datEmbedded bitmap data (not XML, skip)

The inspector's get_symbols() reads every LIBRARY/*.xml (recursing into subfolders) and builds a dict keyed by the symbol's name attribute, not the filename. The name inside the XML is canonical.

  frameRate="2"        ← fps (2 = "on twos", 24fps animation)
  creatorInfo="Adobe Flash Professional CC"
  versionInfo="Saved by Adobe Flash Windows 14.2 build 20"
  xflVersion="2.5">

  <folders>
    <DOMFolderItem name="STOCK .fla stuff" itemID="4b5a..."/>
  </folders>

  <media>
    <DOMBitmapItem             ← embedded image, references bin/*.dat
      name="STOCK .fla stuff/STOCK_bg1"
      href="STOCK .fla stuff/STOCK_bg1.png"
      bitmapDataHRef="Media 34.dat"/>
  </media>

  <symbols>                    ← manifest: lists every library symbol
    <Include
      href="~TS_Character.xml"  ← relative path inside LIBRARY/
      itemID="58bf0560-00000085"
      lastModified="1488913760"/>
    <Include href="TS_Iris.xml" .../>

The inspector only uses the <symbols> section as a sanity check; it reads the actual symbol data directly from the LIBRARY XML files.

LIBRARY/*.xml — Symbol Files

Each symbol is stored as its own XML file. The root element is always <DOMSymbolItem>:

<DOMSymbolItem
  xmlns="http://ns.adobe.com/xfl/2008/"
  name="TS_Iris"              ← this is the canonical key! used in libraryItemName refs
  itemID="58bf0560-000000a8"    ← Flash's internal GUID (ignore)
  symbolType="graphic"        ← "graphic", "movieclip", or "button"
  lastModified="1488913760">  ← Unix timestamp

  <timeline>
    <DOMTimeline name="TS_Iris">
      <layers>
        ...
      </layers>
    </DOMTimeline>
  </timeline>

</DOMSymbolItem>
symbolTypeMeaning
graphicSimple graphic. Playback is controlled by the parent (firstFrame, loop). Most symbols in these FLAs.
movieclipIndependent timeline, plays on its own regardless of parent frame.
button4-frame timeline (Up/Over/Down/Hit). Rare in animation FLAs.

The inspector's get_symbols() keys the dict by the name attribute, not the filename. When a DOMSymbolInstance says libraryItemName="TS_Iris", that's the string you look up in this dict.

Timeline → Layers

Inside every symbol (and on the stage) there is one <DOMTimeline>. It contains a flat list of <DOMLayer> elements:

<DOMTimeline name="~TS_Head">
  <layers>
    <!-- [0] guide layer — not rendered -->
    <DOMLayer
      name="AUDIO"
      layerType="guide"
      visible="false">...</DOMLayer>

    <!-- [20] mask layer — clips layers below it that have parentLayerIndex="20" -->
    <DOMLayer
      name="EYE mask"
      layerType="mask">...</DOMLayer>

    <!-- [21] normal layer — masked by layer [20] -->
    <DOMLayer
      name="HIGHLIGHTS"
      parentLayerIndex="20">...</DOMLayer>

    <!-- [22] another normal layer — also masked by [20] -->
    <DOMLayer
      name="IRIS"
      parentLayerIndex="20">...</DOMLayer>

    <!-- regular layer, no masking -->
    <DOMLayer name="SKULL">...</DOMLayer>
  </layers>
</DOMTimeline>

Layer stacking order

The first layer in the XML list is the topmost layer in Flash's layer panel (rendered last = on top). The last layer in the list is at the back. The inspector reverses the list to draw back-to-front.

Layer attributes

DOMFrame — Keyframes

Each layer has a <frames> block containing a list of <DOMFrame> elements. These are keyframes. Flash's frame model:

  • A keyframe starts at index and lasts for duration frames
  • Frame numbers 0, 1, 2, ... index into this span
  • If you want frame N, find the keyframe where index ≤ N < index + duration
<frames>
  <DOMFrame index="0" duration="100" keyMode="9728">
    <elements/>            ← empty frame (100 frames of nothing)
  </DOMFrame>

  <DOMFrame index="100" duration="32"
            name="Rotation" labelType="comment">
    <!-- name="Rotation" is a frame LABEL (or comment) -->
    <elements>
      ...
    </elements>
  </DOMFrame>

  <!-- last frame — duration omitted! (see bug section) -->
  <DOMFrame index="132">
    <elements>...</elements>
  </DOMFrame>
</frames>

DOMFrame attributes

AttributeMeaning
index0-based start frame number. Always present.
durationHow many frames this keyframe covers. Omitted on the last keyframe — treat as 1 (or extend to infinity; see bugs).
nameFrame label or comment text. Optional.
labelType"name" (default), "comment", "anchor"
tweenType"motion", "shape", or absent (keyframe). The inspector doesn't use tweens.
keyModeInternal Flash hint. Ignore.

Finding the active frame for frame N: The inspector's _active_frame(layer, N) scans all DOMFrames and returns the one whose index ≤ N < index + duration. If no frame matches (frame N is past all keyframes), it returns None. This silently skips symbols placed on the very last keyframe when exporting anything other than frame 0.

DOMSymbolInstance — Placing a Symbol

This is the most common element. It places one symbol from the library onto the stage or into another symbol.

<DOMSymbolInstance
  libraryItemName="TS_IrisWedges"    ← key into symbols dict
  symbolType="graphic"
  loop="loop"                       ← playback mode
  firstFrame="0">                  ← which frame of the symbol to show

  <matrix>
    <Matrix
      a="-0.999984741210938"         ← scale X (with rotation)
      b="0"                          ← shear Y
      c="-0.0987548828125"           ← shear X
      d="0.999969482421875"          ← scale Y (with rotation)
      tx="17.45"                     ← translation X in PIXELS
      ty="11"/>                      ← translation Y in PIXELS
  </matrix>

  <transformationPoint>
    <Point/>                        ← pivot/registration point (usually origin)
  </transformationPoint>

  <color>                          ← optional: color/alpha transform
    <Color alphaMultiplier="0.5"/>
  </color>

</DOMSymbolInstance>

loop attribute

ValueMeaning

playOncePlay once, then hold on the last frame

For the parser, firstFrame is the most important attribute — it tells you which frame of the child symbol to render. The inspector passes it as inst_frame when recursing into the child.

Matrix Transforms & Twips

The 6-component affine matrix

Flash uses a standard 2D affine transform stored as 6 components:

a c tx b d ty scaleX/rot shear/rot translate

Applied to a point (x, y):

x' = a·x + c·y + tx
y' = b·x + d·y + ty

The identity matrix is a=1, b=0, c=0, d=1, tx=0, ty=0. When a <Matrix> element only has tx and ty, the a/b/c/d fields default to identity (1,0,0,1).

⚠ Units: Twips vs Pixels

This is the #1 source of bugs. Flash uses two units:

  • Twips — edge geometry in the XML edge string. 1 twip = 1/20 pixel.
  • Pixels — matrix tx/ty values and stage width/height.

The parser divides all edge coordinates by 20 (parse_edge_str does / 20.0 after decoding). So after parsing, geometry is in pixels. Matrix tx/ty are already in pixels. They should match — but only if no extra ×20 is accidentally applied elsewhere.

WhatUnit in XMLAfter parser
Edge coordinatesTwips (20×px)Pixels (÷20)
Matrix tx/tyPixelsPixels (unchanged)
Stage width/heightPixelsPixels (unchanged)
Gradient matrix tx/tyPixelsPixels (unchanged)
StrokeStyle weightPixelsPixels (unchanged)

Composing nested matrices

When a symbol contains instances which contain more instances, you compose transforms:

# result = parent ∘ child
result_a  = parent_a * child_a  + parent_c * child_b
result_b  = parent_b * child_a  + parent_d * child_b
result_c  = parent_a * child_c  + parent_c * child_d
result_d  = parent_b * child_c  + parent_d * child_d
result_tx = parent_a * child_tx + parent_c * child_ty + parent_tx
result_ty = parent_b * child_tx + parent_d * child_ty + parent_ty

This is what _compose_mat does. The translation columns (tx/ty) mix linearly through the linear part of the parent, so any unit mismatch compounds at every nesting level.

DOMShape — Vector Art

A DOMShape holds three indexed tables and an edge list. Nothing is rendered until you look up the right index.

<DOMShape>

  <!-- Table 1: Fill styles (color, gradient) -->
  <fills>
    <FillStyle index="1">
      <SolidColor color="#A66BBF"/>
    </FillStyle>
    <FillStyle index="2">
      <SolidColor color="#D19FE4"/>
    </FillStyle>
  </fills>

  <!-- Table 2: Stroke styles -->
  <strokes>
    <StrokeStyle index="1">
      <SolidStroke weight="2" caps="round">
        <fill>
          <SolidColor color="#000000"/>
        </fill>
      </SolidStroke>
    </StrokeStyle>
  </strokes>

  <!-- Table 3: Edges (actual path data) -->
  <edges>
    <Edge
      fillStyle0="1"
      fillStyle1="2"
      strokeStyle="1"
      edges="!881 -493.5|881 -603.5..."/>
    <Edge
      fillStyle0="1"
      edges="..."/>
  </edges>

</DOMShape>

Key design insight: fill and stroke indices are local to each DOMShape. Index 1 in one shape has nothing to do with index 1 in another shape.

FillStyle — Colors & Gradients

SolidColor

<FillStyle index="1">
  <SolidColor
    color="#A66BBF"    ← HTML hex color
    alpha="0.75"/>      ← 0-1 opacity (omitted = fully opaque)
</FillStyle>

LinearGradient / RadialGradient

<FillStyle index="1">
  <LinearGradient>           ← or RadialGradient
    <matrix>
      <!-- maps gradient space → shape's local pixel space -->
      <Matrix
        a="-0.0028"              ← gradient is rotated/scaled
        tx="278.85" ty="48.65"/>
    </matrix>
    <GradientEntry color="#210A3F" ratio="0"/>      ← ratio 0 = start
    <GradientEntry color="#672F8B" ratio="1"/>      ← ratio 1 = end
  </LinearGradient>
</FillStyle>

Gradient space: a linear gradient runs from x = -819.2 to x = +819.2 at y = 0. A radial gradient has center (0,0) and radius 819.2. The gradient matrix maps this abstract space into the shape's coordinate system. (819.2 = 16384 twips ÷ 20.)

StrokeStyle

<StrokeStyle index="1">
  <SolidStroke
    weight="2"           ← stroke width in pixels
    caps="round"         ← "round" | "square" | "none" (= butt)
    joints="round">      ← join type (optional)
    <fill>
      <!-- stroke color uses the fill syntax! -->
      <SolidColor color="#000000" alpha="1"/>
    </fill>
  </SolidStroke>
</StrokeStyle>

The stroke color lives inside a <fill> child — confusing but that's how Flash stored it (strokes can technically be gradient-stroked).

Edge String — Path Encoding

The edges attribute of an <Edge> element is a compact path encoding. It looks like gibberish at first but follows a simple grammar.

Commands

TokenCommandArguments
!Move tox y
|Line tox y
[Quadratic beziercx cy ex ey (control point, then end point)
SnStyle selectorn = integer — skip entirely, Flash uses fillStyle0/1 attrs instead

Coordinate encoding

FormatExampleMeaning
Plain integer881881 twips (= 44.05 px after ÷20)
Plain decimal-493.5–493.5 twips (= –24.675 px)
#hex#345Hex twips: 0x345 = 837 (short = positive)
#hex.hex#FFFD68.46Signed fixed-point: int part + frac/256. 6-digit hex ≥ 0x800000 is negative (24-bit two's complement).

Real example — annotated

edges="!354 -261[439 -100 396 90!396 90[352 280 206 389|#FFFF1B.93 #1B0.19"

Token by token:
  ! 354 -261          → MoveTo (354÷20, -261÷20) = (17.7, -13.05)
  [ 439 -100 396 90   → QuadBezier ctrl=(21.95, -5) end=(19.8, 4.5)
  ! 396 90            → MoveTo (19.8, 4.5)   ← Flash restates the endpoint
  [ 352 280 206 389   → QuadBezier ctrl=(17.6, 14) end=(10.3, 19.45)
  | #FFFF1B.93 #1B0.19 → LineTo: hex decode...
       #FFFF1B.93 → int=0xFFFF1B=16776987 ≥ 0x800000 → signed: 16776987-16777216 = -229
                    frac=0x93=147 → 147/256 ≈ 0.574 → value = -228.426
                    in px: -228.426÷20 = -11.42
       #1B0.19    → int=0x1B0=432, frac=0x19=25 → 432.097÷20 = 21.60

Redundant MoveTo after curves: Flash always restates the current endpoint as a new MoveTo after every quadratic bezier. The parser suppresses these (they're within 0.015px of the current position) to avoid creating disconnected path segments. This is the cmds_to_svg_d epsilon check.

The cubics attribute

Some Edge elements also have a cubics attribute. This is Flash's cubic bezier hint data — it's for the visual editor only and is not used by the parser. Ignore it completely when parsing.

Decoding #hex yourself

  1. Split on . — left is integer part, right is fractional (or absent = 0)
  2. Parse integer hex: if it's ≥ 6 digits AND ≥ 0x800000 → subtract 0x1000000 (signed)
  3. Parse fraction hex: value is frac / 256.0
  4. Add integer + fraction, then divide total by 20 to get pixels

Fill Sides — fillStyle0 vs fillStyle1

This is the most subtle part of Flash's vector model. Each Edge has up to three style references:

<Edge
  fillStyle0="1"     ← fill on the LEFT  as you travel the edge direction
  fillStyle1="2"     ← fill on the RIGHT as you travel the edge direction
  strokeStyle="1"    ← stroke drawn on top of the edge
  edges="..."/>

Flash stores shape boundaries this way because a single edge segment can be the boundary between two different filled regions simultaneously. The edge knows which fill is on each side.

How the parser handles this

fillStyle0 edges — used as-is. The fill is on the left of the travel direction, which corresponds to "fill inside" for counter-clockwise contours (SVG's fill-rule="evenodd" default).

fillStyle1 edges — reversed! To paint the fill that's on the right, we flip the edge direction so the fill becomes the left side. The parser calls _reverse_seg() on each segment before adding it to the fill's segment list.

# Conceptually:
for each Edge:
    if fillStyle0:
        fill_segs[f0].extend(segments)          # forward direction
    if fillStyle1:
        fill_segs[f1].extend(reverse(segments)) # reversed!
    if strokeStyle:
        stroke_segs[ss].extend(segments)        # direction irrelevant for stroke

Known bug: _reverse_seg quadratic control point — When reversing a quadratic bezier, the control point should be "reflected" to maintain the same curve shape. The current code keeps the original control point, which produces a geometrically different curve. This is only visible on curved fillStyle1 edges. See the bugs section.

Layer Types & Masking

Layer types

layerTypeBehaviour
normal (or absent)Regular layer. Renders normally.
guideMotion guide or reference layer. Not rendered. Skip entirely.
folderOrganizational folder — contains no frames itself. Skip.
maskClips the layers that reference it via parentLayerIndex.

How masking works

<!-- Layer list (0-indexed): -->
[20] <DOMLayer name="EYE mask"   layerType="mask">  ← MASK layer
[21] <DOMLayer name="HIGHLIGHTS" parentLayerIndex="20">  ← clipped by [20]
[22] <DOMLayer name="IRIS"        parentLayerIndex="20">  ← clipped by [20]

The parser renders this as an SVG <mask>:

  1. Render the mask layer geometry as all-white shapes → goes into <defs><mask id="mask_N">...
  2. Render the masked layers normally inside a <g mask="url(#mask_N)">

Mask layers themselves are rendered all white (via _shape_svg_white) because SVG masks use luminance: white = fully visible, black = fully clipped. The actual colors on the mask layer don't matter for the clip shape.

The mask layer's parentLayerIndex is NOT set — only the masked layers have this attribute, pointing up to their mask. So to find all layers clipped by mask layer [20], you scan all layers for parentLayerIndex="20".

Color Transforms on Instances

A DOMSymbolInstance can have an optional <color><Color .../></color> child that modifies every pixel rendered by the instance:

<color>
  <Color
    redMultiplier="1"       ← multiply channel 0-1 (default 1)
    greenMultiplier="0.5"
    blueMultiplier="0"
    alphaMultiplier="0.75"  ← opacity multiplier (most common usage)
    redOffset="50"          ← add to channel after multiply, -255..255
    greenOffset="0"
    blueOffset="0"
    alphaOffset="0"/>
</color>

The formula is: out = in * multiplier + offset/255 per channel.

The inspector currently only looks at alphaMultiplier and only indirectly — color transforms aren't propagated through the SVG rendering (SVG opacity on the <g> would be the fix). If you see semi-transparent elements rendering fully opaque, this is why.

Gradients in Detail

Gradient coordinate space

Flash defines a "gradient box" in abstract coordinates:

  • Linear: runs from (-16384, 0) to (+16384, 0) in twips = (-819.2, 0) to (819.2, 0) in pixels
  • Radial: center (0, 0), radius 16384 twips = 819.2 pixels

The gradient's <matrix> transforms this box into the shape's local pixel coordinate system. So if you see a gradient that looks like it's in the wrong place, the matrix is where to look.

GradientEntry

AttributeRangeMeaning
color#RRGGBB hexStop color
ratio0.0 – 1.0Position along the gradient axis
alpha0.0 – 1.0Opacity at this stop (omitted = 1.0)
<GradientEntry color="#210A3F" ratio="0"/>      ← left/inner edge
<GradientEntry color="#552A79" ratio="0.247"/>
<GradientEntry color="#672F8B" ratio="1"/>      ← right/outer edge

Ratios don't have to be evenly spaced. Flash (and SVG stop offset) both allow any 0–1 ordering.

⚠ Known Parser Bugs

These are documented in REFACTOR.md. Listed from least to most risky to fix:

Bug 1 — Last keyframe duration

Flash omits the duration attribute on the last keyframe of a layer. The parser defaults it to 1, so any frame index beyond lastKeyframe.index + 1 returns None (layer disappears). For frame-0 exports this doesn't matter, but for any multi-frame export, characters placed on long-running final keyframes will vanish.

<!-- last frame — duration is MISSING, not "1" -->
<DOMFrame index="200"> ← extends to end of symbol's timeline, not just frame 201
  <elements>...</elements>
</DOMFrame>

Fix: if no frame matches frame N, return the last frame whose index <= N.

Bug 2 — _reverse_seg quadratic control point

When reversing a bezier segment for fillStyle1 rendering, the control point should be reflected geometrically. The current code keeps the original control point, producing a slightly different curve shape.

# Wrong (current): control point stays at same position
result.append(('Q', c[1], c[2], sx, sy))   # c[1],c[2] is old control point

# Correct reversed quadratic bezier:
# start at end, the control point stays exactly the same (because
# a quadratic bezier P0→ctrl→P1 reversed is P1→ctrl→P0)
# Actually the current code IS correct for quadratics! The control
# point of a quadratic doesn't change when you reverse direction.
# The endpoint does change (sx,sy is the original start). ✓

Wait — actually quadratic reversal IS correct. For a quadratic bezier P0→ctrl→P1, the reversed curve P1→ctrl→P0 uses the same control point. The REFACTOR.md notes it as high-risk to touch, and the math checks out as correct. This may be a false alarm — but verify visually on curved fillStyle1 edges.

Bug 3 — Segment chain tie-breaking (_segs_to_svg_d)

When multiple segments share a start point (adjacent filled regions meeting at a shared vertex), the stitcher picks cands[0] — the first in list order, which is arbitrary. This can produce wrong contour closures for complex shapes with shared vertices.

In _segs_to_svg_d:
cands = [i for i in start_idx.get(rk(ex, ey), []) if i not in visited]
nxt = cands[0]   ← picks arbitrarily when multiple segments start here

Fix: sort candidates by angle relative to incoming direction (pick the
one that continues most "straight ahead" or by winding order).

Bug 4 — Unit mismatch under matrix composition

Edge geometry and matrix tx/ty are both in pixels after parsing — they should be compatible. But the REFACTOR.md documents a historical issue where the wrong unit was used, causing nested symbol translations to compound incorrectly. If you see nested symbols shifted relative to each other, check whether any code path is accidentally ×20-ing the tx/ty somewhere.

Quick test: a symbol with two levels of nesting.
If the leaf-level art is offset by exactly 20× the expected amount,
that's the unit bug. If it's just slightly off, it's probably a
different matrix composition issue.

Bug 5 — Mask geometry size

SVG masks need a width/height that covers all masked content. The current hardcoded values (x="-500" y="-500" width="5000" height="5000") work for most symbols but will silently clip symbols that extend beyond this range.

<mask id="mask_TS_Head_20" maskUnits="userSpaceOnUse"
      x="-500" y="-500" width="5000" height="5000">  ← hardcoded!
Fix: compute bbox of masked content and use that + padding.
